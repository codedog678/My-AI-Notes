# task01

## 1.工具和技能

### 🛠 工具（Tools）

- **定义**：外部的功能接口或系统能力。
- **作用**：提供具体的操作能力，比如访问文件系统、运行 Bash 命令、调用搜索引擎、连接数据库。
- **特点**：通常是底层的、单一功能的，直接执行某个动作。
- **类比**：就像一把锤子或螺丝刀，能完成某个具体操作。

### 🎯 技能（Skills）

- **定义**：在智能体框架中，基于工具和指令组合而成的“能力模块”。
- **作用**：把工具调用、逻辑流程、上下文管理封装成一个可复用的任务。内部可能调用多个工具和指令，形成一个完整的工作流。
- **特点**：更高层次，面向任务，通常包含多个步骤和逻辑。
- **类比**：就像“修一张桌子”的完整技能，里面可能用到锤子、螺丝刀、尺子等工具。

> 当你有一个**反复要求智能体实现的工作流**时，与其每次都解释相同的工作流，不如将其打包成一个技能，让智能体自动知道该做什么。

**工具流（Tool Flow）**

- 工具之间的组合调用顺序。
- 技能往往就是在工具流的基础上，加上逻辑、上下文和封装。
- 可以理解为：**工具流是技能的底层实现，技能是工具流的对外接口**。

## 2.skills文件夹

（1）指令（Instructions）

- **是什么**：描述技能或任务的规则、流程和调用方式。
- **作用**：告诉智能体“该怎么做”，比如输入是什么、输出是什么、调用顺序如何。
- **怎么起作用**：
  - 定义技能的接口（参数、返回值）。
  - 规定执行逻辑（例如：先搜索 → 再分析 → 最后生成报告）。
  - 相当于“操作手册”，让智能体知道如何使用技能。

（2） **脚本（Scripts）**

- **是什么**：具体的代码实现，通常是 Python、Bash 或其他语言写的。
- **作用**：真正执行任务的底层逻辑。
- **怎么起作用**：
  - 当智能体调用某个技能时，脚本会运行，完成具体操作。
  - 例如：一个“数据清洗技能”的脚本可能包含 Pandas 代码来处理 CSV 文件。
  - 相当于“工具的程序化实现”，是技能的执行引擎。

（3） **资产与资源（Assets and Resources）**

- **是什么**：技能运行所需的外部文件或依赖，比如图片、模板、数据集、配置文件。
- **作用**：为技能提供支持材料。
- **怎么起作用**：
  - 模板：比如生成报告时的 Word 模板。
  - 数据：比如预训练好的模型文件或参考资料。
  - 配置：比如 API 密钥、环境变量。
  - 相当于“材料库”，让技能在执行时有东西可用。

##  3.渐进式披露YAML+md+元数据| 指令|资源

 **“渐进式披露（Progressive Disclosure）”** 部分，主要讲的是如何通过 **YAML + Markdown + 元数据** 来组织和加载技能的内容。它的核心思想是：不同层次的信息在不同阶段被加载，既保证了灵活性，又避免了冗余。

> **元数据**：始终存在，描述技能的身份和基本信息。
>
> **资源**：按需调用，提供技能执行的支持材料。

+ YAML 的基本语法示例

```yaml
# 一个简单的 YAML 配置示例
name: Agent Skills
version: 1.0
enabled: true

tools:
  - bash
  - search_web

skills:
  - name: MarketResearch
    description: Collect and analyze market data
    steps:
      - search_web
      - process_data
      - generate_report
```

>  解释：

- `name: Agent Skills` → 键值对
- `tools:` → 列表（缩进的 `-` 表示数组元素）
- `skills:` → 嵌套结构，里面包含多个字段（name、description、steps）

> 🔗 在智能体/技能开发中的作用

- **指令部分**：常用 YAML 来定义技能的输入输出、参数和调用方式。
- **脚本部分**：脚本本身可能是 Python/Bash，但它的配置和调用规则往往写在 YAML 里。
- **资产与资源部分**：YAML 可以用来描述资源路径、依赖关系。

> YAML+md+元数据

1. **YAML**

- **位置**：通常在文件开头的 *frontmatter* 部分。
- **作用**：用来写结构化的配置，比如技能名称、描述、参数、标签。
- **特点**：机器可读，方便智能体快速识别和调用。

2. **元数据（Metadata）**

- **位置**：往往就是写在 YAML 里。
- **作用**：描述技能的“身份信息”，比如作者、版本、用途。
- **特点**：属于 YAML 的内容，但强调的是“信息属性”，而不是执行逻辑。

3. **Markdown（md）**

- **位置**：在 YAML 之后的正文部分。
- **作用**：写技能的指令、执行步骤、说明文档。
- **特点**：人类可读，适合写详细的操作流程和解释。

因为 **元数据通常写在 YAML 里**，所以“YAML”和“元数据”常常重叠。Markdown 部分也可能包含一些“说明性元数据”（比如输入输出格式），所以和 YAML 的作用有交叉。在实际使用中，三者往往放在同一个文件里，形成一个整体：

- YAML（结构化配置）
- 元数据（身份信息，通常在 YAML 内）
- Markdown（详细说明和指令）

```markdown
技能文件夹
│
├── YAML 前置部分（Frontmatter）
│     ├── 元数据（Metadata）
│     │     ├── name: 技能名称
│     │     ├── description: 技能描述
│     │     ├── version: 版本号
│     │     └── tags: 标签/分类
│     └── 配置参数（Parameters）
│           ├── 输入/输出定义
│           ├── 工具调用规则
│           └── 环境变量
│
├── Markdown 正文（Instructions）
│     ├── 技能的详细说明
│     ├── 执行步骤（流程）
│     ├── 输入输出示例
│     └── 注意事项/解释
│
└── 资源（Resources）
      ├── 模板文件（如 Excel、Word）
      ├── 示例数据（如 CSV）
      ├── 配置文件（如 API 密钥）
      └── 参考文档/规则
```

## 4.实践建议与最佳实践

- 明确 Skill 的输入输出标准，示例文件放在 references 目录
- 所有脚本应有异常处理与错误报告能力，便于 Agent 自动修复
- 复杂逻辑建议分模块实现，主流程在 SKILL.md 中清晰描述
- Excel 公式相关操作建议分离脚本处理，避免直接在 openpyxl 中计算
- 尽量输出中间结果与最终数据，便于人工或 Agent 二次校验

## 5.通用型 Agent：代码是通用接口

**Simple Scaffolding: bash and filesystem**

- 意思是：最简单的 Agent 脚手架就是能调用 **命令行（bash）** 和 **文件系统**。
- 这是最基础的工具接口，能让 Agent 执行文件操作、运行脚本。

**But they need context and domain expertise to do the job reliably**

- 意思是：仅仅有 bash 和文件系统还不够。
- Agent 要想可靠地完成任务，还需要 **上下文（context）** 和 **领域专业知识（domain expertise）**。
- 这些就需要通过 **Skills** 来补充。

>  Agent 本身不再被限定为某个领域的专家，而是一个 **通用框架**。它通过 **Skills** 来获得领域能力
>
> **过去**：Agent 本身就是领域专家（功能写死在 Agent 内部）。
>
> **现在**：Agent 是通用工匠，领域能力由外部的 **技能文件夹（YAML + md + 资源）** 提供。
>
> **结果**：一个通用 Agent 可以在不同场景下灵活切换，只要加载对应的技能即可。

==**Agent 本身只是一个通用框架，如果没有 Skills，它的能力非常有限。**==

- **有了 Skills**：
  - Agent 可以快速加载领域知识（比如法律审查流程）。
  - 可以复用标准化的工作流程（比如每周营销复盘）。
  - 可以获得新的能力（比如生成报告或搭建服务器）。
- **没有 Skills**：
  - 每次使用 Agent 都要从零开始描述任务。
  - 每次都要手动提供参考资料。
  - 输出结果可能不一致，缺乏稳定性。
